# 微博归档器 v0.2.0 产品需求文档（PRD）

## 项目信息

- **Language**: 中文
- **Programming Language**: Express + Playwright + Axios（后端），React 18 CDN + Babel Standalone（前端）
- **Project Name**: weibo_archiver
- **版本**: v0.2.0
- **原始需求**: 为微博归档器增加强制全量爬取入口、全局搜索、图片下载、帖子分页加载、订阅列表分页排序、Git 仓库配置等功能

---

## 产品定义

### Product Goals

1. **数据完整性保障**：确保用户能够随时对任何订阅进行全量重新抓取，不受增量逻辑限制，消除数据遗漏风险
2. **高效内容检索**：提供灵活的多维度搜索能力，让用户在大量归档数据中快速定位目标帖子
3. **浏览体验优化**：通过分页加载、图片下载、列表排序等交互改进，提升大规模归档数据的浏览与管理效率

### User Stories

**强制全量入口**
- As a 归档用户, I want 在订阅管理界面强制触发全量爬取 so that 即使之前只抓了部分数据，也能补全所有历史帖子
- As a 归档用户, I want 在全量爬取前看到确认提示 so that 避免误操作导致不必要的长时间抓取

**全局搜索 + 指定订阅搜索**
- As a 归档用户, I want 在一个搜索框中输入关键字并选择搜索范围 so that 可以在所有订阅或指定用户的帖子中快速找到内容
- As a 归档用户, I want 按时间范围、有图/无图条件过滤搜索结果 so that 能精确定位特定时期或特定类型的帖子

**图片下载按钮**
- As a 归档用户, I want 在查看帖子时点击按钮将图片保存到本地 so that 可以离线浏览和备份喜欢的图片

**分页加载**
- As a 归档用户, I want 自定义每页显示的帖子条数 so that 根据我的屏幕和阅读习惯调整浏览密度
- As a 归档用户, I want 直接输入页码跳转到指定页 so that 不需要反复翻页就能快速到达目标位置

**订阅列表分页 + 排序**
- As a 归档用户, I want 订阅列表按用户名字母排序 so that 能快速找到特定订阅
- As a 归档用户, I want 订阅列表也支持分页显示和页码跳转 so that 订阅数量多时不会一次性加载过长列表

**Git 仓库配置**
- As a 开发者, I want 项目有完善的 README 和仓库配置 so that 其他用户能快速了解并使用该项目

## UI Design Draft

### 1. 强制全量爬取入口

- **位置**：订阅卡片操作区域，在现有「抓取」按钮旁新增「全量抓取」按钮
- **样式**：使用醒目的橙色/红色调以区分普通抓取按钮
- **交互**：点击后弹出确认弹窗，内容：「即将对 [用户名] 进行全量抓取，可能耗时较长，是否继续？」，确认后触发全量抓取
- **状态**：抓取进行中按钮显示 loading 状态，进度信息通过 SSE 推送展示

### 2. 搜索功能

- **位置**：页面顶部新增搜索栏区域
- **布局**：
  - 左侧：搜索输入框（单行文本，placeholder：「搜索帖子内容…」）
  - 右侧：搜索范围下拉选择（全部订阅 / 指定用户——选择后弹出用户选择器）
  - 输入框下方展开：高级筛选条件（时间范围选择器 + 有图/无图筛选按钮）
- **结果展示**：搜索结果替换帖子列表区域，匹配关键字高亮；顶部显示结果总数和当前筛选条件标签

### 3. 图片下载按钮

- **位置**：帖子图片右上角，hover 时显示下载图标
- **交互**：点击即触发浏览器下载该图片文件到本地
- **文件名**：保留原始文件名或使用「{用户名}_{帖子ID}_{序号}.jpg」格式

### 4. 帖子分页加载

- **位置**：帖子列表底部
- **布局**：
  - 左侧：每页条数选择器（下拉框：10/20/30/40/50）
  - 中间：分页控件（上一页 / 页码 / 下一页）
  - 右侧：页码跳转输入框 + 跳转按钮；显示「第 X / Y 页」
- **交互**：切换每页条数后自动重新加载第一页；输入页码回车或点击跳转按钮跳转

### 5. 订阅列表分页 + 排序

- **排序**：订阅列表默认按用户名拼音/字母升序排列
- **分页**：与帖子分页控件样式一致，默认 10 条/页，可选至 50 条/页
- **位置**：订阅列表区域底部

## 待确认问题

1. **强制全量抓取的数据处理策略**：全量抓取时，已存在的帖子是否覆盖更新？还是仅补全缺失帖子？——建议：覆盖更新（以最新数据为准），需确认
2. **搜索 API 性能考量**：当前数据存储为本地 JSON 文件，全文搜索在大数据量下的性能如何？是否需要引入索引机制？——建议 v0.2.0 先做基础实现，性能问题后续优化
3. **图片下载路径**：是否需要让用户自定义下载目录？还是统一使用浏览器默认下载路径？——建议默认浏览器下载路径，v0.3.0 再考虑自定义路径
4. **搜索范围选择交互**：指定用户搜索时，用户选择器是下拉多选还是弹窗多选？——建议下拉多选，操作更轻量
5. **订阅列表排序维度**：是否需要支持多种排序方式（如按最近抓取时间、帖子数量等）？——建议 v0.2.0 仅支持用户名字母排序，后续扩展
6. **分页参数持久化**：用户选择的每页条数是否需要持久化（localStorage）？——建议持久化，提升体验
